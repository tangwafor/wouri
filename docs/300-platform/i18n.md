# Platform: internationalization

Traces to the plan (FR-LOC-02) and the fleet i18n standard. No em-dashes.

## Two separate language axes
1. **Interface language.** French default, English available, one-tap switch. Every user-facing string ships fr + en with full diacritics, legible at small sizes on a photocopy and a fax. Pidgin and other locales can be added as registry-driven labels later.
2. **Document output language.** The document engine outputs in whichever language the receiving authority requires, INDEPENDENT of the user's interface language. A phytosanitary certificate to a French authority renders in French; a document for a German customs desk renders in the language that desk expects. This is a hard rule, not a preference.

## Rules
- No user-facing string is hardcoded; all come from message catalogues (fr, en) or registry labels for domain terms.
- Registry-driven labels (capabilities, document types, species, requirements) carry label_fr and label_en columns, so a new commodity or rule ships its own translations as data.
- The naming is bilingual with French primary in the product (Dossier d'expedition, Unite d'origine, Points bloquants).
- Never serialise the entire message catalogue into a page that also carries tenant data (a Bazah lesson: next-intl inlines the full catalogue).
- An i18n check gates the build: every key present in every locale, no missing translations.

## Numbers, dates, currency
Money is integer minor units rendered per the display currency with its symbol; dates in the locale's format; the document engine uses the receiving authority's conventions. FX and quantity follow the units and money doctrine (integer base units, null-not-one fx).
